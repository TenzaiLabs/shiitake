Shiitake

Sandboxing for untrusted commands on Kubernetes.

An HTTP server that fans commands out to a pool of resident, resource-bounded worker containers in your Kubernetes cluster. Each command runs in its own worker, which resets to a clean slate between commands — no state bleed, ever.

🚀 Quickstart 📖 Read the docs ★ GitHub

The problem

Running untrusted commands is hard

Lots of systems need to run short-lived, untrusted commands for someone else — user-submitted code, AI-agent tool calls, CI steps, scraped build scripts. Doing it safely is awkward.

🩸

State bleeds between commands

Run them in your own process or container and one command's leftover processes, temp files, or a fork bomb leak into the next caller — or take the whole service down.

🔑

Pod-per-command is heavy

Spawning a Job or Pod per task means handing your workload Kubernetes pod-create rights and paying per-task scheduling latency every single time.

💧

Output overruns memory

A chatty command whose output you buffer in memory turns into an out-of-memory kill of your service, not its own.

How Shiitake solves it

A fixed pool of resident, resource-bounded worker containers behind one authenticated HTTP server. You POST a command and get a handle back; the server hands it to an idle worker that runs it in isolation, streams output straight to disk, and resets to a clean slate before the next command. No pod-create rights, no per-task scheduling latency, no state bleed — and a runaway command can only take down its own worker.

Quickstart

Deploy a sandbox, run a command

A Shiitake Pod — one server, one worker — on any Kubernetes cluster, then a command dispatched over HTTP. Built images only; no cargo required. Works on a local kind or k3d cluster.

1 Build a runnable worker image

The published shiitake-worker image is just the static binary (a scratch image), so pair it with a base that has bash — commands run as bash -c — plus whatever tools your commands need. 

# a minimal worker: bash + the published Shiitake worker binary
cat > Dockerfile <<'EOF'
FROM alpine:3
RUN apk add --no-cache bash
COPY --from=ghcr.io/tenzailabs/shiitake-worker:latest \
     /usr/local/bin/shiitake-worker /usr/local/bin/shiitake-worker
ENTRYPOINT ["/usr/local/bin/shiitake-worker"]
EOF

docker build -t shiitake-worker-demo:latest .
kind load docker-image shiitake-worker-demo:latest   # k3d: k3d image import …

2 Apply the smallest possible deployment

One Pod, two containers — the server (the only ingress) and one worker — sharing a capture volume. That's the whole footprint. 

# shiitake.yaml
apiVersion: apps/v1
kind: Deployment
metadata: { name: shiitake }
spec:
  replicas: 1
  selector: { matchLabels: { app: shiitake } }
  template:
    metadata: { labels: { app: shiitake } }
    spec:
      containers:
        # the control plane — accepts HTTP, dispatches to workers, serves output
        - name: server
          image: ghcr.io/tenzailabs/shiitake-server:latest
          env:
            - { name: SHIITAKE_AUTH_TOKEN, value: dev-secret-token }
            - { name: SHIITAKE_CAPTURE_ROOT, value: /capture }
          ports: [{ containerPort: 8080 }]
          volumeMounts: [{ name: capture, mountPath: /capture }]
        # one resident worker; reaches the server on 127.0.0.1 (shared pod netns)
        - name: worker
          image: shiitake-worker-demo:latest
          imagePullPolicy: IfNotPresent
          env:
            - { name: SHIITAKE_WORKER_ID, value: worker-0 }
            - { name: SHIITAKE_CAPTURE_ROOT, value: /capture }
          volumeMounts: [{ name: capture, mountPath: /capture }]
      # SHIITAKE_CAPTURE_ROOT is a SHARED VOLUME mounted into both containers at the
      # same path: the worker writes each command's stdout/stderr here, the server
      # reads it back over HTTP. An emptyDir lives and dies with the Pod.
      volumes: [{ name: capture, emptyDir: {} }]
kubectl apply -f shiitake.yaml
kubectl rollout status deploy/shiitake

Minimal on purpose. OOM detection (needs pod-read RBAC), per-worker recycling, and scaling the pool are all in the Kubernetes docs.

3 Reach the API

# the server is the only ingress; forward its port to your machine
kubectl port-forward deploy/shiitake 8080:8080

4 Dispatch a command and read it back

# POST a command — get a handle back immediately (202 Accepted)
curl -s -H "Authorization: Bearer dev-secret-token" \
  -X POST localhost:8080/api/v1/exec \
  -d '{"command": "echo hello from a sandboxed worker"}'
# → {"handle":"01J…","started_at":"…"}

# read the captured stdout for that handle
curl -s -H "Authorization: Bearer dev-secret-token" \
  localhost:8080/api/v1/exec/01J…/stdout
# → hello from a sandboxed worker

echo is a bash builtin, so it needs nothing else. A command that calls an external binary must carry a PATH in the request's env — the worker runs with a cleared environment. See the API reference.

How it works

One command, there and back

Follow a single command through the system — from your POST, into an isolated worker, and all the way back to you as captured output.

request path response path — back to you

  1. You POST the command

    The client sends POST /api/v1/exec with the command and an Authorization: Bearer token to the server — the single, authenticated ingress. Workers never face the network.

  2. The server schedules it

    It picks an idle worker from the resident pool and hands the command over an in-pod WebSocket on loopback. If every worker is busy you get 429 — Shiitake never silently queues behind a slow command.

  3. You get a handle, immediately ↩ to you

    The server replies 202 Accepted with a { handle }. You don't hold a connection open while the command runs — everything after this is polled against that handle.

  4. The worker runs it

    The worker executes the command as bash -c in its own process group, inside a resource-bounded container it can't escape. No other command's processes, files, or environment are visible to it.

  5. Output streams to the shared volume

    stdout/stderr file descriptors are redirected straight into per-handle capture files on the shared capture volume. The kernel writes to disk, so neither process buffers output in memory — gigabytes of output is fine.

  6. The worker reports and resets

    On exit it reports the exit code, cause, and byte counts back to the server, which marks the handle done. The worker then sweeps leftover processes, clears scratch, and re-advertises itself idle — recycling its container if a reset can't be trusted clean.

  7. You read the result back ↩ to you

    Poll GET /api/v1/exec/{handle} for status, then GET …/stdout (and /stderr). The server serves the captured bytes from the shared volume with HTTP Range support, so you can tail the end or resume a partial read.

What it's for

Built for hostile input

Shiitake is generic — it knows nothing about your application. A few things it's particularly good at:

  • 🛡️ Sandbox untrusted commands

    Every command runs in its own resource-bounded worker container — it can't see or touch another command's process, files, or output. The worker resets to a clean slate between commands (and recycles a fresh container if it can't), so nothing carries over to the next caller.

  • ⚡ Run commands without pod-create rights

    Skip giving your workload Kubernetes permissions to spawn a Job or Pod per task. A fixed pool of workers is already running and waiting — you just POST a command and get a handle back. No RBAC to create pods, no per-task scheduling latency.

  • 💥 OOM-killed protection

    Memory limits are enforced at the container level, so an allocation bomb only kills its own worker — the server and every other worker keep running. The kill is detected from the kubelet's container status and reported as oom_container, never silently lost.

  • 🌊 Handle large outputs

    The worker redirects stdout/stderr straight into capture files via inherited fds — the kernel writes to disk, so neither process buffers output in memory. Read it back over HTTP with Range support, and watch output size as a metric instead of truncating it.
